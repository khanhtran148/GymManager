### Summary

The most systemic quality issue is that all five handlers call `.ToString()` on domain error objects before passing them to `Result.Failure`, which likely breaks structured error-to-HTTP-status mapping in `HandleResult`. Closely related is a cross-slice coupling violation where two handlers directly reference the `ToDto` helper defined inside `RecordTransactionCommandHandler`; the mapping logic should live in the shared `TransactionDto` or a dedicated mapper. The frontend has one data-integrity concern: a synthesised revenue chart presents fabricated weekly bucketing as real data, and a churn-rate double-multiplication bug produces a display value 100x too large.

### Unresolved Questions

- Does `ApiControllerBase.HandleResult()` pattern-match on the error *type* or on the error *message string*? If it matches on the string content, the `.ToString()` calls may accidentally work, but the design is still fragile and should be corrected.
- Is `GetExpenseAggregateAsync` intentionally reserved for a future slice, or can it be removed now?
