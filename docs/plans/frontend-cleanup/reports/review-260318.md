## Critical / High Findings

### 1. [HIGH — Security] RbacProvider has no error fallback — infinite loading screen
- **File:** `src/apps/gymmanager-web/src/components/rbac-provider.tsx:32-34`
- **Issue:** If `GET /roles/metadata` fails permanently, the provider renders an infinite loading skeleton, blocking the entire dashboard.
- **Fix:** Check `isError` from `useRolesMetadata()` and render an error fallback or fall back to defaults.

### 2. [HIGH — Performance] Middleware parses base64 + JSON on every request
- **File:** `src/apps/gymmanager-web/src/middleware.ts:47-68`
- **Issue:** `parseRouteAccessCookie` runs `atob` + `JSON.parse` + per-element validation on every authenticated page navigation.
- **Fix:** Cache the parsed result at module scope, keyed by raw cookie value.

### 3. [HIGH — Performance] `permission-toggle-grid` does O(P*R) linear scans per render
- **File:** `src/apps/gymmanager-web/src/components/permission-toggle-grid.tsx:193-205`
- **Issue:** `getRoleBitmask` does `Array.find` per cell (100 scans for 20 perms * 5 roles).
- **Fix:** Pre-index `roleData` into `Record<roleName, bigint>` with `useMemo`.

### 4. [HIGH — TFD] 4 new production modules have zero test coverage
- **Files:** `rbac-store.ts`, `rbac-provider.tsx`, `use-roles-metadata.ts`, `middleware.ts` (parseRouteAccessCookie)
- **Issue:** Direct TFD violation — these shipped without tests.
- **Fix:** Write tests for all 4 before merging.

## Security Notes (Acknowledged — Not New)

The `is_authenticated` and `user_role` cookies are client-writable JS cookies. The middleware route guard is **UX-only** (backend `IPermissionChecker` is the security boundary). The `GET /roles/metadata` endpoint exposes structural RBAC metadata to all authenticated users — this is intentional per the API contract (UX metadata, not tenant-specific permission assignments). These are pre-existing architectural decisions, not regressions from this change.
